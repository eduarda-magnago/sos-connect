## Controle de Versão

A ferramenta de controle de versão adotada no projeto foi o
Git, sendo que o Github
foi utilizado para hospedagem do repositório.

<img width="707" height="434" alt="Gitflow" src="https://github.com/user-attachments/assets/1442a3b5-ed89-4ef2-afd0-18f758aa92f6" />

O projeto segue a seguinte convenção para o nome de branches:

- `main`: versão estável já testada do software
- `unstable`: versão já testada do software, porém instável
- `testing`: versão em testes do software
- `dev`: versão de desenvolvimento do software

Quanto à gerência de issues, o projeto adota a seguinte convenção para
etiquetas:

- `documentation`: melhorias ou acréscimos à documentação
- `bug`: uma funcionalidade encontra-se com problemas
- `enhancement`: uma funcionalidade precisa ser melhorada
- `feature`: uma nova funcionalidade precisa ser introduzida


## Gerenciamento de Projeto

### Divisão de Papéis

Apresente a divisão de papéis entre os membros do grupo.

Exemplificação: A equipe utiliza metodologias ágeis, tendo escolhido o Scrum como base para definição do processo de desenvolvimento. A equipe está organizada da seguinte maneira:
- Scrum Master: Carolina;
- Product Owner: Eduarda;
- Equipe de Desenvolvimento: Carolina, Eduarda, Eryc, Igor, Lucy, Maressa, Maya;
- Equipe de Design: .

*Para organização e distribuição das tarefas do projeto, a equipe utilizará o Projects do Github*

- `Backlog`: Lista de tarefas que ainda não foram iniciadas, apenas planejadas ou ideais futuras.
- `Ready`: Tarefas que foram priorizadas e estão prontas para serem iniciadas.
- `In Progress`: Tarefas que estão em andamento.
- `In Review`: Tarefas concluídas que precisam passar por testes ou revisão antes de serem finalizadas.
- `Done`: Tarefas finalizadas, aprovadas e entregues.

### Processo

-  `Backlog`
  
Product Owner cria e prioriza as tarefas do Backlog, registrando no Projects, define o que precisa ser feito e detalha requisitos básicos para a equipe entender.

- `Ready`
  
O Scrum Master facilita a definição das tarefas do sprint. A equipe de desenvolvimento e a de design selecionam e se preparam para executar as tarefas, garantindo que tudo esteja claro e viável.

- `In Progress`
  
As equipes executam as tarefas. Desenvolvimento codifica ou implementa; design cria layouts, interfaces ou materiais visuais. O Scrum Master acompanha impedimentos e ajuda a remover obstáculos.

- `In Review`
  
As equipem validam e testam as tarefas entregues. Desenvolvedores testam funcionalidades; designers revisam interfaces e consistência visual. Scrum Master garante que os testes estejam dentro das definições.

- `Done`
  
Após a equipe confirmar que a tarefa está concluída, testada e aprovada. O Product Owner valida se atende aos requisitos; Scrum Master garante que os critérios de qualidade foram cumpridos.

## Ferramentas

As ferramentas empregadas neste projeto são:

- Visual Studio Code

O Visual Studio Code foi selecionado por sua estabilidade, integração com o Windows e o GitHub, e pelo suporte de uma ampla comunidade de usuários que disponibiliza plugins, extensões, tutoriais e correções de bugs, aprimorando a experiência da ferramenta.

- Microsoft Teams

O Microsoft Teams foi selecionado por centralizar a comunicação e a colaboração em tempo real, integrando reuniões, chats e gerenciamento de arquivos.

- Whatsapp

WhatsApp foi escolhido para a comunicação periódica da equipe, como forma de contato direto e acessível, considerando seu amplo uso.

- Figma

  Figma foi escolhido por sua capacidade de colaboração em tempo real, suporte a grandes arquivos de forma online, interação com interfaces navegáveis, simulando previamente o processo de desenvolvimento, e acesso pelo navegador, sem a necessidade de instalação de programas.

- Draw.io

  O draw.io foi selecionado por sua facilidade de uso, compatibilidade com múltiplas plataformas e integração com serviços de armazenamento na nuvem, permitindo a criação e compartilhamento eficiente de diagramas.
